# Debug Trace Protocol（调试推演协议）

逐步调试推演的标准化格式。像 debugger 单步调试一样，用触发 bug 的具体数据逐步推演每一个方法的执行过程。

---

## 第一步：声明触发数据

明确声明能触发 bug 的全部上下文数据：

```
══════════════════════════════════════════════════════
TRIGGER DATA（触发数据）
══════════════════════════════════════════════════════
入参:     {userId: 123, action: "activate"}
数据库:   users(id=123, status="deleted", name=null)
外部依赖: paymentService.check(123) → {active: true}
══════════════════════════════════════════════════════
```

## 第二步：逐步推演（核心）

沿着调用链，**每个关键执行节点画一个 STEP 框**，框内严格包含五个字段。步骤之间用 `▼` 箭头连接。

### STEP 框格式（五个字段缺一不可）

```
┌─ STEP {N} ── {类名.方法名()} ───────────────────┐
│ 输入: {进入此步时的变量/参数状态}                   │
│ 执行: {此步执行的具体逻辑}                         │
│ 输出: {执行后的结果/返回值/变量新状态}              │
│ 预期: {应该得到什么} ── 实际: {实际得到什么} ── ✅  │
└──────────────────┬───────────────────────────────┘
                   ▼
```

### 判定行规则

- 符合预期 → `预期: xxx ── 实际: xxx ── ✅`
- 不符合预期 → `预期: xxx ── 实际: xxx ── ❌`
- ❌ 的步骤后紧跟 `★ 根因:` 说明

### 完整示例

```
══════════════════════════════════════════════════════
DEBUG TRACE（逐步调试推演）
══════════════════════════════════════════════════════

┌─ STEP 1 ── EntryController.handle() ────────────┐
│ 输入: request = {userId: 123, action: "activate"} │
│ 执行: 解析请求参数，提取 userId=123                │
│      调用 userBizService.activate(123)            │
│ 输出: 参数 userId=123 传递给下游                   │
│ 预期: 参数正确传递 ── 实际: 参数正确 ── ✅         │
└──────────────────┬───────────────────────────────┘
                   ▼
┌─ STEP 2 ── UserService.getById() ───────────────┐
│ 输入: userId = 123                                │
│ 执行: SELECT * FROM users WHERE id = 123          │
│ 输出: user = {id:123, status:"deleted", name:null}│
│ 预期: 返回用户对象 ── 实际: 返回非null对象 ── ✅   │
└──────────────────┬───────────────────────────────┘
                   ▼
┌─ STEP 3 ── 条件分支判断 ────────────────────────┐
│ 输入: user.status = "deleted"                     │
│ 执行: 判断 user.getStatus() == "active"           │
│ 输出: false → 走 NO 分支 → 进入激活逻辑           │
│ 预期: 应进入激活逻辑 ── 实际: 进入激活逻辑 ── ✅   │
└──────────────────┬───────────────────────────────┘
                   ▼
┌─ STEP 4 ── UserBizService.activate() ───────────┐
│ 输入: user.name = null                            │
│ 执行: userName = user.getName() → null            │
│      log.info("激活: " + userName.toUpperCase())  │
│ 输出: NullPointerException 异常抛出               │
│ 预期: 正常打印日志 ── 实际: NPE异常 ── ❌          │
│                                                   │
│ ★ 根因: name 为 null 时未做空值检查，直接调用      │
│   null.toUpperCase() 导致 NPE                     │
└──────────────────────────────────────────────────┘
```

## 协议规则

| 规则 | 说明 |
|------|------|
| **STEP 编号连续** | 从 STEP 1 开始，每步 +1，不跳号 |
| **五字段完整** | 每个 STEP 必须包含：输入、执行、输出、预期/实际/判定。缺一不可 |
| **判定行必填** | 每个 STEP 最后一行必须是 `预期: xxx ── 实际: xxx ── ✅/❌` |
| **条件分支独立成步** | `if/switch` 判断单独作为一个 STEP，输出是"走了哪个分支" |
| **方法调用展开** | 关键方法调用展开为独立 STEP；与 bug 无关的简单方法可合并到调用方 STEP 内 |
| **❌ 后必跟根因** | 判定为 ❌ 的 STEP 内，紧跟 `★ 根因:` 说明 |
| **数据必须具体** | 所有变量值必须是具体数据（如 `null`、`123`），禁止用"某个值"等模糊描述 |
| **箭头连接** | STEP 之间用 `▼` 箭头连接，表示执行顺序 |

## 第三步：根因结论卡

在推演图下方，输出精确的根因结论卡：

```
┌─ ★ ROOT CAUSE ──────────────────────────────────┐
│ 文件:       UserBizService.java                   │
│ 方法:       activate()                            │
│ 行号:       第 47 行                              │
│ 触发条件:   user.getName() 返回 null              │
│ 错误:       调用 null.toUpperCase() 抛出 NPE      │
│ 为什么现在才出: 之前用户 name 有默认值，本次导入   │
│              的用户 name 为 null                   │
└──────────────────────────────────────────────────┘
```

## 修复后对比推演格式

修复方案验证时，使用以下对比格式：

```
══════════════════════════════════════════════════════
DEBUG TRACE ── 修复前 vs 修复后
══════════════════════════════════════════════════════
使用触发数据: {userId: 123, action: "activate"}
══════════════════════════════════════════════════════

────── 修复前（引用定位推演结果）──────

STEP 4 ── UserBizService.activate()
  输入: user.name = null
  执行: userName.toUpperCase()
  输出: NullPointerException
  预期: 正常打印日志 ── 实际: NPE异常 ── ❌

────── 修复后 ──────

┌─ STEP 4 ── UserBizService.activate() ───────────┐
│ 输入: user.name = null                            │
│ 执行: userName = user.getName() → null            │
│      检查 userName == null → true                 │
│      抛出 BizException("用户名不能为空")          │
│ 输出: BizException 业务异常，调用方捕获处理       │
│ 预期: 空值安全处理 ── 实际: 抛出业务异常 ── ✅    │
└──────────────────────────────────────────────────┘
```

**对比推演规则**：
- 修复前：简要引用 ❌ 的 STEP，不需要重画全部步骤
- 修复后：从发生变化的 STEP 开始，用完整 STEP 框格式画出修复后的执行过程
- 每个 STEP 同样必须包含五字段
- 修复后所有 STEP 的判定应为 ✅
- 至少用 1 组边界数据推演修复后的代码，验证无副作用
