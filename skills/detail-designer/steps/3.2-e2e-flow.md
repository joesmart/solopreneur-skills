# Step 3.2: End-to-End Flow Design

**Apply the ReAct protocol from SKILL.md to execute this step.**

## Action

Draw ASCII flow diagrams using the standardized node protocol. This is the core blueprint of the design.

**MANDATORY**: Before drawing flows, load [references/output-template.md](../references/output-template.md) to understand the complete node protocol format and visual indicators for changes.

**Node protocol format** (every node MUST include all of these):
```
┌──────────────────────────────────────────────────────────────────┐
│ [ServiceName]: [LayerName] [Status: ✨ New/🔧 Extend/📝 Modify]  │
├──────────────────────────────────────────────────────────────────┤
│  [ID] ClassName                                                  │
│      File: full file path                                        │
│      Method: methodName(paramTypes) → returnType                 │
│      Responsibility: one sentence describing what this node does │
│      Input: specific input data structure                        │
│      Output: specific output data structure                      │
│      │                                                           │
│      ├── Step 1: specific operation description                  │
│      ├── Step 2: specific operation [🆕 新增逻辑] (if new/modified) │
│      └── Step 3: specific operation [🆕 新增逻辑] (if new/modified) │
└────────────────────────┬─────────────────────────────────────────┘
                         │ [Communication: HTTP POST /path | Method call | MQ]
                         ▼
```

**Visual indicators for changes** (CRITICAL for highligng new/modified logic):
- Use emoji status: ✨ New, 🔧 Extend, 📝 Modify
- Mark new/modified steps with [🆕 新增逻辑] or [修改逻辑]
- This makes changes immediately visible to reviewers

**Cross-service boundaries MUST annotate:**
- Service name
- Communication protocol (HTTP/RPC/MQ)
- Interface path
- Data format (JSON/encrypted/compressed)

## Checklist

Use these items in your **[CHECK]** stage:

- Every node has complete 8-item information (ID/File/Method/Responsibility/Input/Output/Steps/Status)
- All cross-service boundaries annotated with protocol and data format
- Data flow from entry to storage is complete with no gaps
- At least 1 exception handling branch included
- Flow covers all scenarios from Phase 1 understanding
- ✨ New/🔧 Extend/📝 Modify status clearly marked with emoji in node headers
- 🆕 New or modified steps marked with [🆕 新增逻辑] or [修改逻辑] tags for visibility

## Challenges

Use these questions in your **[CHALLENGE]** stage:

- Q1: What breaks this? → If developer uses only this flow to code, where would they get stuck?
- Q2: What didn't I check? → Is any node's Input/Output vague (like "user info" instead of specific fields)?
- Q3: Expert criticism? → If system architect reviews this flow, would they think it's detailed enough?
